Shared Subroutines Patch

Created and updated by imamelia and supported by HammerBrother.

What this patch does is enable you to have “universal availability” subroutines accessible anywhere in your ROM, and also by other third-party patches, tools, code without the need to have duplicate codes that can waste space in your game. The reason for this patch is because patches commonly need a freespace (in the olden days when xkas was popular, you have to manually set them up, and Asar automatically finds those freespace), thus the code themselves can have varying locations each time they are inserted and isn't safe (if other resources calls a subroutine at an outdated address, may cause glitches or crashes).

It works like this: The game stores a list of JMLs at a fixed location. Each of these JMLs point to subroutines in a freespace. When calling routines, any code from anywhere can JSL to any of the JMLs on the list, which then jumps to the freespace subroutine. If you make changes on the subroutines so that they get relocated on the next re-insert, the JML list will be corrected and remain at the same location (provided you do not re-patch this with the !Freespace_SharedSub_JMLList being changed).

^How to insert

  1. First thing off, make sure you have a Lunar Magic restore point or backup of your game file before patching this. It's possible you might set the freespace address of the JML list incorrectly or not have enough space and cause your game to crash when running it. If this happens, you can revert your game file to before this occurs and then try patching again with a different address to use.
  2. Open SharedSub_Defines/SharedSubroutineDefs.asm, and here, you see a define !Freespace_SharedSub_JMLList, this is the address to manually place your JML list (comes with an 8-byte RATS tag at this location before the list if needed). This must be at a freespace with enough bytes available and at a starting address location of that (it occupies this address and consecutive N bytes after).

      What's important here is the LoROM Offset and the size. LoROM Offset is the starting SNES memory address location that's empty, and size is the number of bytes free. Now knowing which one having enough space is important. The amount of bytes we need is: 
      BytesTaken = RatsSpace + (NumberOfJMLs * 4)

;RatsSpace:
; = 0 if you have !Freespace_SharedSub_JMLList be at banks $00-$0F (or $80-$8F using LoROM)
; = 8 otherwise ($10-$7F (or $90-$FF using LoROM)).

      In case you don't know, a bank is a byte of the leftmost (or highest) 8-bit value of a 24-bit/3-byte address: the XX in $XXyyyy.

      With the example above showing freespace, I choose $128000 (so I set !Freespace_SharedSub_JMLList to that address number), now look at the left two hex digits, that is bank $12, and if you haven't added any new routines or remove any, the number of bytes the JML list takes is 208 = 8 + (50 * 4). With $128000 having $8000 (32768) bytes free, that is enough space to fit the 208-byte JML list in. This results in $128000 to $1280CF now being occupied. Upon patching sharedsub.asm, asar will display the memory statistics about the JML list (including the RATs tag when needed).

    • To locate freespace without slogger (especially if you're using SA-1 ROM, which may have issues), another method is to create a dummy ASM file with the following text and patching with Asar so that it searches freespace with enough space for you (tested on Asar version 1.91) and showing the address: 
      !NumberOfJMLs = 100
	;^The number of JMLs you expect to have. No prefix is decimal.

	;Auto calculate the amount of bytes needed
		!NumberOfBytesNeeded #= !NumberOfJMLs*4
freecode
Test:
	fillbyte $00
	fill !NumberOfBytesNeeded
	print "Freespace with enough bytes is at $", hex(Test-8)
		;^Minus 8 because the address defined as !Freespace_SharedSub_JMLList would be the beginning of the RATS tag. Stuff under
		; "freecode" would be placed at the beginning of the freespace plus 8.
		;
		; This also means that asar looks for a freespace that is at least DataSizeInFreespace+8, where DataSizeInFreespace is
		; the amount of bytes of anything under "freecode", and +8 because it includes the RATS tag.

      Once asar runs this, it will display the print command text with the address you can use (copy that), along with a warning that “This freespace appears to be leaked” and questions if you want to insert anyway. Answer “no” (else you'll waste freespace) to not modify the ROM.

      If you get an error saying “No freespace found”, that means there are no freespace with sufficient bytes available. If this happens, either don't patch Shared Subroutines, or remove some other ASM stuff or data and try again.
  3. Open subroutinecode.asm and on the very bottom after the last subroutine, create your routine, here is a simple example: 
    MyRoutine123:
	;Sets current player's lives to 99.
	LDA.b #98 ;>In SMW's lives system, the value stored in memory is -1 from displayed (if HUD says 5, its #$04)
	STA $0DBE|!addr
	RTL
  4. Next, create an autoclean JML RoutineMainLabel on the JML list entry like so: 
    ;[...]
autoclean JML LoseYoshi				; make Yoshi run away
autoclean JML MyRoutine123			;>This new routine
JMLListEnd: ;>The byte-address +1 from the last byte.
;^Don't have items after this label, otherwise stuff past here is not protected by RATS, and Asar information display will be off.
    Now run Asar to insert the routines package into your game.

    If you get an error saying “Don't autoclean a label at the end of a freespace block” error message, that means a contiguous sequence of code in subroutinecode.asm is too large to fit in a freespace block. This can be solved by adding the text “freecode” in between the subroutines just before it goes past the end of the freespace block. This causes asar to find a different freespace location for the code right after it to be placed at, thus separating the code section.

    However, don't put it in between subroutines that are connected via branch (BEQ/BNE/BPL/BMI/BCC/BCS/BVC/BVS/BRA/BRL) or a 2-byte jump (JMP $xxxx): 
    Subroutine1:
	;<Some_code_here>
	BEQ Subroutine2_Label1
	
	;<Some_code_here>
	JMP Subroutine2_Label2

freecode

	;Can be any number of subroutines here, and is still bad to have freecode here
	;because you have jumps to a location that may be in a different bank, or is
	;outside the branch's bounds (128 bytes back from the address of the opcode
	;after the branch, or 127 bytes forward from such opcode).

Subroutine2:
	;<Some_code_here>
	.Label1
		;<Some_code_here>
	.Label2
		;<Some_code_here>
	RTL
    If you want to port your codes that uses !defines naively (or defines used exclusively in the routine) to the shared subroutines patch (many patches uses subroutines), see this information. on how to do so.

    Note that there may be some subroutines included you don't need in your hack, if that's the case, feel free to remove the JML list, the define associated to, and the subroutine itself to save space. Make sure the order between the JML list and the macro define list must match.

^How to use

  1. To use routines in Shared Subroutines from third-party resources, like from blocks (GPS), sprites (pixi), patches (asar), uberasm tool, etc, you must copy the folder SharedSub_Defines and include SharedSubroutineDefs.asm using the incsrc "<Path_to_define_file>" command at the top of the asm files that calls the subroutinecode.asm's routines. Then to call it, simply do JSL !SubroutineName: 
    incsrc "SharedSub_Defines/SharedSubroutineDefs.asm"
	;[...]
	JSL !MyRoutine123  ;> An example of a code in a third-party ASM resource calling a subroutine in Shared Subroutines.
  2. After that, just insert by running the respective tool.

^How to relocate JML list

If you have inserted Shared Subroutines and need to relocate the JML list to another area in ROM or just remove Shared Subroutines entirely, make sure you follow these instructions.

  1. Whatever you do, do not just simply change !Freespace_SharedSub_JMLList and re-patch. This only spawns a new JML list without removing the old one.
  2. Prerequisites:
    • Define !Freespace_SharedSub_JMLList (in SharedSub_Defines/SharedSubroutineDefs.asm) should not be modified. This is also used as a target location to clear the JML list.
    • The define list (in SharedSub_Defines/SharedSubroutineDefs.asm) and the JML list (in sharedsub.asm) should not be modified. Otherwise the subroutine codes would not be properly cleared.
    • In addition to above, the defines and JML list must have equal number of items. Otherwise it might clear an incorrect number of bytes of the JML list.
  3. Patch SharedSubRemover.asm to your game. This will remove Shared Subroutines entirely. If you just want to remove that, Then the following instructions should be ignored. In that case, you should fix any third-party code in your game that still uses Shared Subroutines, as they now point to a non-existent code that crashes your game.
  4. Open SharedSub_Defines/SharedSubroutineDefs.asm and change the address in !Freespace_SharedSub_JMLList to some other address. Then repatch sharedsub.asm.
  5. In any ASM resources using Shared Subroutines, make sure you have updated all copies of define file SharedSubroutineDefs.asm and reinsert the ASM codes.

^Workings documentation

Each subroutine holds these items:
  • Label ↔ definition files:
    • sharedsub.asm which holds the JML list (under the JMLListStart: label), Format: autoclean JML SubroutineName

    • SharedSubroutineDefs.asm contains definitions regarding the location of the JMLs and a macro that generates new defines for each JML list item. Because Asar (or other patching tools) is unaware of what is installed inside the ROM, so it cannot obtain the locations of the start of the routines (not without using a large number of read<Number>(<Address>) for asar).

    Be aware that the order of the JML list in sharedsub.asm and the macro defines list in SharedSubroutineDefs.asm must match (this includes not to have missing defines on a JML item that does exist, and vice versa), for example: 
    ;[...]
autoclean JML SubroutineName
autoclean JML AnotherSubroutineExample

    means that in the definition file, it needs to be ordered like this: 
    ;[...]
%SetSharedSubDefine(SubroutineName)
%SetSharedSubDefine(AnotherSubroutineExample)

    and not this: 
    ;[...]
%SetSharedSubDefine(AnotherSubroutineExample)	;>Gets assigned to [SubroutineName]
%SetSharedSubDefine(SubroutineName)		;>Gets assigned to [AnotherSubroutineExample]

    And this: 
    autoclean JML FindFreeUploadSlot
autoclean JML GetRand2
autoclean JML RangedRandomRt ;>Not assigned to define.
autoclean JML ChangeMap16
autoclean JML FindMap16ActsLike
autoclean JML FindMap16TileNum
autoclean JML SubGetItemMemory
;[...]
    %SetSharedSubDefine(!FindFreeUploadSlot)
%SetSharedSubDefine(GetRand2)
%SetSharedSubDefine(ChangeMap16)		;>Gets assigned to [RangedRandomRt]
%SetSharedSubDefine(FindMap16ActsLike)		;>Gets assigned to [ChangeMap16]
%SetSharedSubDefine(FindMap16TileNum)		;>Gets assigned to [FindMap16ActsLike]
%SetSharedSubDefine(SubGetItemMemory)		;>Gets assigned to [FindMap16TileNum]
;[...]
    Because then the labels gets assigned to the wrong define(s).

    What actually happens is that a define gets assigned to an address, the address is then increased by 4 (because JML occupies 4 bytes), then assign the next define at that new address, and repeats for each define. It does not regard the actual names of the subroutine labels corresponding to the define names; only regarded by guessing based on their list positions/“index”. This also means you can rename the defines to avoid potential define conflicts (e.g. %SetSharedSubDefine(LoseYoshi) can be changed to %SetSharedSubDefine(SharedSub_LoseYoshi) to form the define !SharedSub_LoseYoshi and it will still work).

    One thing you should avoid is setting the same define again (duplicate %SetSharedSubDefine(DefineSubroutineName) in the define list), as every copy will re-assign the address value of the same define. The JML list manager linked will also detect that.

  • subroutinecode.asm holds the subroutine code itself. Format: 
    SubroutineName:
<Code here>
;ends using an RTL, not RTS.

  • I've provided a JavaScript tool that manages the list of subroutines.

Notes

  • Be careful to avoid having label duplicates (i.e “redefined label”) in the subroutine codes, as asar would error out saying that they are redefined (you cannot have a label point to 2 or more locations). This likely happens when you insert a newer version of a subroutine in addition to the old one, or add a new routine that does not use sublabels. To avoid this, edit the code to use sublabels (number of “.”s before the name of the sublabel), use +/- (have them repeated should one jumps over another (example: BRA ++ to branch downards to the first ++)), rename the main labels, or remove the old and outdated subroutine.

  • It is safe to call a routine within a routine, since they are in the same patch (Asar knows the labels within subroutinecode.asm, so you won't get error saying “label not found”) because multiples files connected via incsrc), you can simply call the subroutine directly without using a define (JSL SubroutineName instead of JSL !SubroutineName to save a few cycles, since you don't have to jump to a JML in the JML list, then back into the routine code)

  • Due to how the SNES bank borders work, if you have a very large code in subroutinecode.asm and/or a large JML list, Asar will error out saying a bank border have been crossed. To fix this, have freecode near the area (between two subroutines, data, or table, not during that area) that is about to cross the border. This will tell asar to have the following code in a different freespace address location.

    In LoROM, banks $00~$7D (according to the image in Ersanio's ASM tutorial) reserves the lower half ($0000~$7FFF, 32768 bytes) for other things (hardware registers, RAM mirrors, etc.). The upper half of the bank ($8000~$FFFF, 32768 bytes) is ROM data, which is where you place your code, including the JML list, in. With only 32768 bytes available in each section, you can have up to 8192 JMLs, or 8190 JMLs should you have a RATS tag applied. To calculate the maximum JMLs you can have in a range of addresses, the formula is MaxJMLs = floor((MaxAddress - MinAddress + 1 - RatsSpace) / 4). Here's a calculator you can use:

  • The game may glitch/crash if you patch Shared Subroutines (step 1), then a third-party patch that uses freespace (step 2), then re-apply Shared Subroutines with a longer list of JMLs (step 3) upon executing code of the third party patch:

    That is because the previous patch at step 2 must've been placed a freespace code after the JML list, then gets overwritten by the expanded JML list upon reapplying. To avoid this in the first place, before patching Shared Subroutines the first time, create placeholder JMLs (they're just empty routines that just exist so that in the future, you can replace it to be an actual routine) on the list so that other freespace patches don't “cut in line” of the JML list, here is what I mean: 
    ;Make sure the list ends with nth number of these, and not to have actual routines after if you're not going to have these defined,
;else you would assign the defines to the wrong label.
autoclean JML Placeholder	;\Take up space so that future 3rd-party patches will not be placed here (reserve that space)
autoclean JML Placeholder	;|
autoclean JML Placeholder	;|
autoclean JML Placeholder	;|
autoclean JML Placeholder	;|
;[...] ;>expand this as many as you would expect how many routines you would need.
JMLListEnd: ;>The byte-address +1 from the last byte.

    The result:

    subroutinecode.asm: 
    PlaceHolder:
	RTL	;>A blank subroutine that just exist (you only need to have this code exist once).
		;You can also just point them to the same subroutine (any will work).
    If you do run into this type of issue, best way to handle this is to restore your game to a previous state (in Lunar Magic's menu bar, it's under File → Restore → Restore ROM to previous state...) that is before your first time of inserting Shared Subroutines patch.

  • To add to above, if you re-patch Shared Subroutines with a bigger list of JMLs but the amount it increases is no bigger than 8 bytes (equivalent to 2 JMLs), then only the third-party patch's RATS tag at the beginning of its freespace gets corrupted (RATs tags are 8-bytes long) while the code itself is safe to execute. When this happens, problem may arises the next time you insert any freespace-type patch (they end up thinking the third-party's code is freespace, especially given that the first 4 bytes “STAR” are now garbage values).
  • The provided routines in this ASM file may be out of date. Newer tool features and such can happen long after the last update of this ASM resource. Feel free to update them if you wish.
  • The information displayed on the Asar console window when patching is whats actually inserted into the ROM, not based on defines. So if you have more items in the defines than on the JML list, that won't do anything to the game, nor the display (you probably want the number of entries in both lists to be equal anyways).
  • If you run into define conflicts (two ASM resources uses of the same define for something different) on an ASM resource using Shared Subroutines, you can restore the defines' value by including the define file again (incsrc) at the spot where you want it to be restored (does not have to be at the top of the ASM file).
