^Updating defines

Perhaps you want to move this simple example code to the shared subroutines patch: 
!Define0 = $60
!Define1 = $61
!Define2 = $62
!Define3 = $63


MyRoutine123:
	LDA #$01
	STA !Define0
	LDA #$02
	STA !Define1
	LDA #$03
	STA !Define2
	LDA #$04
	STA !Define3
	RTL
Pretty simple! just copy the whole thing, making sure nothing unrelated modifies the defines in between whats sets them (top) and the codes that uses them (this happens if 2 or more patches happens to have defines with the same name, like for example, player HP is defined !Health = $60 and a sprite using the same define, !Health = $7C). Of course, the same applies if the defines are in a separate file and uses incsrc "<path>" (includes “source file”)

defines.asm: 
!Define0 = $60
!Define1 = $61
!Define2 = $62
!Define3 = $63


AnotherDefinesFile.asm: 
!AnotherDefine0 = $7C
!AnotherDefine1 = $87

Routine code: 
incsrc "defines.asm"
incsrc "AnotherDefinesFile.asm" ;>In the case you have an ASM resource with defines being in different ASM files.
MyRoutine123:
	LDA #$01
	STA !Define0
	LDA #$02
	STA !Define1
	LDA #$03
	STA !Define2
	LDA #$04
	STA !Define3
	
	LDA !AnotherDefine0
	CLC
	ADC #$05
	STA !AnotherDefine0
	
	LDA !AnotherDefine1
	SEC
	SBC #$05
	STA !AnotherDefine1
	RTL

If you have another subroutine that uses the same defines as the other, place the defines or the “include source” before ALL (on top) of them so you don't get errors.

However, not all patches have conveniently named defines, resulting variables in them to be altered when they get redefined, often when two unrelated codes use the same define, for example: just !Health is set to be a value of $60, another and later !Health is set to $0F3A. As asar process the ASM files in a sequential analysis, !Health will contain a value of whats last sets it every time. With such a name, this is bad as that the naming term could be used for completely different purposes (is this for enemy or player? Better to use name terms like !RAM_PlayerHealth and !RAM_EnemyHealth. To fix this issue, its best to edit their defines in their original ASM file (and in the defines file if present), then move them to here (so you don't accidentally edit other existing codes in the shared patch). If multiple ASM files and/or resources intentionally use the same define name (often with the same default value), such as the ground pound patch and a block in the blocks section that interacts with ground pound, make sure ALL of them are renamed for a more specific term.

Renaming the defines can easily be done using a text editor of your choice (I recommend using notepad++) and replace (CTRL+H), then carefully replace all of them by pressing “replace” (I recommend replacing one at a time and paying attention to how it is highlighted, not to be recommended to “replace all” for a fast edit because the “find” mechanism may find sequence of characters within a different define, like searching for “!Freeram_test” could point to !Freeram_test2 when you are looking for just !Freeram_test).

^Fix labels

Subroutines found in ASM files in pixi, Gopher Popcorn Stew (GPS), uberasm tool, and others often uses macros to call them. And because of label leaks, they need to use ?MacroLabel and ?.MacroSubLabels. In these cases, simply copy the entire code, paste them in a temporary blank text area, and use notepad++'s (or any text editor with such a feature) replace feature (CTRL+H) with its search mode set to “Regular Expression” and do this:

Find: 

\?(\.*[a-zA-Z\d]+)
Replace with (all of them, but make sure you check what's being found in case it matches something you don't intend to replace): 
\1

Then copy that converted labels code, and paste it in the shared subroutine's code.
